# plan-site skill

You're building a **report site**: a small, self-contained, multi-page static HTML site that lives in a `plans/` folder and explains or proposes something to a human, who reads it in a browser. It is the heavier-weight sibling of two other skills — reach for the right one:

- **`plan`** → a markdown `.plan.md` spec that *agents* pass between each other. Terse, skimmable, for review-before-build.
- **`ui-mockups`** → one self-contained HTML page that compares UI *options* for a single design decision.
- **`plan-site` (this)** → a multi-page HTML *site* that **explains or proposes a whole system** — prose + diagrams + tables + evidence, with a sidebar to move between pages. Use it when the thing to communicate is too big and too visual for a markdown file, and the audience is a human reading for understanding, not an agent parsing for execution.

Think: "an architecture explainer", "a redesign proposal across six screens", "a caching/performance audit with diagrams", "an onboarding walkthrough of how subsystem X works". The output is a folder the user opens with `./serve.sh` and reads like a little internal microsite.

## The deliverable

A site folder, fully standalone (no network at view time — Tailwind is vendored), plus a **single shared dashboard server** for the whole project that hosts it:

```
plans/
  _server/              # ONE dashboard server for the WHOLE project (shared; see below)
    plans_server.py     #   dynamic index + markdown renderer (stdlib only)
    ensure.sh           #   start-or-reuse the project server under pm2
    .gitignore          #   auto-created by ensure.sh — keeps .port/__pycache__ out of git
  YYYY-MM-DD-<slug>/     # this report site
    index.html          # page 1 — the overview / thesis
    <page-2>.html       # subsequent pages, one concern each
    <page-3>.html
    site-nav.js         # THE OUTLINE: declares pages/groups ONCE → the sidebar is derived
    meta.json           # {title, description, date} — what the dashboard shows for this site
    assets/
      tailwind.css      # vendored Tailwind v4 build (regenerated by build-css.sh)
      report.css        # the report "chrome" — the Spine sidebar + component vocabulary
      report.js         # renders the Spine nav from site-nav.js; scroll-spy, theme, drawer/sheet, quick-jump, debug overlay
    build-css.sh        # rescans the .html pages → rewrites assets/tailwind.css
    serve.py            # no-cache static server — a quick standalone look at THIS site only
    serve.sh            # ./serve.sh [port]
    README.md           # what this is, how to view, the page list, provenance
```

- `YYYY-MM-DD` from `date +%Y-%m-%d` (day created; keep fixed on later edits).
- Slug: kebab-case, ≤5 words, names the subject (`pwa-caching`, `allocator-proposal`, `auth-rework`).
- If the project keeps plans somewhere other than `plans/`, mirror that convention (the `_server/` and `meta.json` rules still apply — put `_server/` at the root of wherever the plans live).

## The project dashboard (one server, not one-per-site)

There is **one** dashboard server per project, living at `plans/_server/`, and it hosts **every** plan-site *and* every markdown `.plan.md` in the project on a single stable port. Do **not** start a pm2 server per site — that old pattern wastes a process and a port on each report. Instead each new site registers with the shared server.

- **It is the index** — discovery is dynamic. The server rescans `plans/` on every request, so a new site or plan appears on a plain refresh; there is no index file to regenerate or keep in sync.
- **It shows every kind and lets you search/sort/filter.** Plan-sites (dated dirs with an `index.html`), markdown plans (`*.plan.md`, read straight from their frontmatter + `> TL;DR`), and UI mocks all appear as cards, with live search, sort (date / title / status / type), and a filter toggle (sites · plans · mocks · all). Markdown plans render in-browser in the same theme; `*.plan-status.md` ledgers are linked from their plan's card.
- **Metadata comes from `meta.json`** for sites (falls back to scraping the page `<title>`/lede for legacy sites, so older sites still show up). Always write `meta.json` for a new site so its card is accurate.
- **It also surfaces the project's UI mocks** (from the `ui-mockups` skill, which live *outside* `plans/`). The server auto-detects the mocks dir (`web/public/mocks`, `frontend/public/mocks`, `public/mocks`, `docs/mocks`, or `mocks` — or set `_server/config.json` to `{"mocksDir": "<path-relative-to-project>"}`), serves it at `/_mocks/`, adds a **Mocks ↗** link to the mocks index, and lists each mock as a card with its lifecycle badge (open / chosen / implemented / cancelled) scraped from the mocks `index.html`. Because the dashboard is the persistent pm2 server, the mocks stay reachable even when the app's own dev server is down.
- It's copied into the project (self-contained, stdlib-only) so you can customize it per project; re-copying to refresh it is safe.

## Step 1 — scaffold from the bundled template

This skill **bundles its own copy** of every asset and script (in the skill's `assets/` and `template/` directories, alongside this SKILL.md). It depends on nothing outside itself. Copy them into the new folder:

```bash
SKILL_DIR="$(dirname "$(readlink -f "$0")")"   # or wherever this skill resolved
DEST="plans/$(date +%Y-%m-%d)-<slug>"
mkdir -p "$DEST/assets"
cp "$SKILL_DIR"/assets/report.css "$SKILL_DIR"/assets/report.js   "$DEST/assets/"
cp "$SKILL_DIR"/template/tailwind.css                              "$DEST/assets/"
cp "$SKILL_DIR"/template/build-css.sh "$SKILL_DIR"/template/serve.py "$SKILL_DIR"/template/serve.sh "$DEST/"
cp "$SKILL_DIR"/template/site-nav.js "$DEST/"            # the outline — edit it (see Step 2)
cp "$SKILL_DIR"/template/page.html  "$DEST/index.html"   # then duplicate per page
cp "$SKILL_DIR"/template/README.md  "$DEST/README.md"
chmod +x "$DEST"/serve.sh "$DEST"/serve.py "$DEST"/build-css.sh

# This site's dashboard card metadata (edit the title/description to the real thing):
cat > "$DEST/meta.json" <<JSON
{"title":"<Report Title>","description":"<one-line what this site explains/proposes>","date":"$(date +%Y-%m-%d)"}
JSON

# Install the project-wide dashboard server ONCE (idempotent — safe to re-copy to refresh it):
PLANS_ROOT="$(dirname "$DEST")"          # the plans/ dir
mkdir -p "$PLANS_ROOT/_server"
cp "$SKILL_DIR"/server/plans_server.py "$SKILL_DIR"/server/ensure.sh "$SKILL_DIR"/server/README.md "$PLANS_ROOT/_server/"
chmod +x "$PLANS_ROOT/_server/ensure.sh" "$PLANS_ROOT/_server/plans_server.py"
```

(If you can't resolve `$SKILL_DIR` programmatically, the files sit next to this SKILL.md — copy them from there.) `template/tailwind.css` is a pre-built base so the report renders immediately; you regenerate it for real in Step 4.

## Step 2 — the page anatomy (the nav is DERIVED, you don't hand-write it)

The TOC is the **"Spine"**: one sidebar tree, rendered by `report.js` from a single manifest. Page numbers, group headers, and the current-page mark are **derived** — never hand-typed, so they can't drift or mis-number across pages. You write two things and the chrome builds itself:

**1. The outline, once, in `site-nav.js`.** Declare the report title, a 2–3 char logo, an optional tagline, and the pages grouped into parts. Order = reading order = numbering:

```js
window.SITE_NAV = {
  report: "Caching Rework", logo: "CR", tagline: "Web · 14 Jun 2026",
  groups: [
    { label: null,          pages: [ { file: "index.html", title: "Overview" } ] },
    { label: "Mechanism",   pages: [ { file: "sw.html", title: "Service worker" },
                                     { file: "store.html", title: "The cache store" } ] },
    { label: "Rollout",     pages: [ { file: "plan.html", title: "Migration plan" } ] },
  ],
};
```

Insert or reorder a page → edit this one file; every page renumbers itself on reload.

**2. Each page's sections, as `id` + `data-toc` on the section elements.** `report.js` reads them from the page body, so the "on this page" list can never disagree with the actual content:

```html
<header  id="overview" data-toc="Overview" …>…</header>
<section id="store"    data-toc="The cache store" … class="… reveal">…</section>
```

Everything else is automatic and identical on every page, all wired by `report.js`:

- **The Spine sidebar** (`[data-nav]`) — the page tree on one rail; the **current page expands in place** to show its `data-toc` sections nested beneath it, with a violet marker that slides to the active section (scroll-spy) while the sub-rail fills with scroll progress. The brand (`[data-brand]`) and a "Page n of N" meter (`[data-meter]`) render from the manifest.
- **Robust scroll-spy** — active section = the last one past a line 30% down the viewport, clamped at both ends, so exactly one is always lit (including page top & bottom). Sets `data-active` + `aria-current`.
- **Press-`/` quick-jump** (`[data-jump]`) — a fuzzy palette over every page and section. Invaluable on large reports.
- **Theme toggle** (`data-theme-toggle`) — persisted light/dark, respects `prefers-color-scheme`.
- **Tablet tier (768–1023px)** — the same sidebar, narrower and **persistent** (no hamburger — this is the old dead-zone, fixed).
- **Mobile tier (<768px)** — a sticky bar shows *On this page · {section}* live with a 2px progress hairline; tap it for a bottom sheet of sections, or ☰ for the whole tree as a focus-trapped drawer.
- **`<main data-main>`** — a `<header>` (kicker + headline + lede) then `<section class="reveal">` blocks. `reveal` fades each section up (disabled under `prefers-reduced-motion`).

Use the editorial layout the template establishes: generous `px-6 sm:px-10 lg:px-14 py-12` section padding, `border-b` between sections, `prose-measure` (≈68ch) on body copy, a small uppercase violet **kicker** above each headline. (The shell/sidebar layout lives in `report.css` as `.ps-*`; your content sections keep using Tailwind utilities.)

## Step 3 — the component vocabulary (in `report.css`)

`report.css` is the report's reusable palette — lean on it instead of inventing styles, so pages stay visually consistent:

| Class | For |
|---|---|
| `.prose-measure` | cap reading width on prose blocks (~68ch) |
| `.reveal` / `.reveal.in` | fade-up a section as it enters the viewport (auto via `report.js`) |
| `.evi` | monospace `file:line` evidence chip — **pin every factual claim to a source** |
| `.statnum` / `.delta` | tabular-nums figures; big delta pills (`−48%`, `74×`) |
| `.mbar` + `.fill-before` / `.fill-after` / `.fill-neutral` + `.lab` | before/after metric bars (a track with a proportional fill) |
| `.sev` + `.sev-high` / `.sev-med` / `.sev-low` | severity/priority dots |
| `.verified` | green "confirmed" emphasis for audit findings |
| `.axis` + `.axis-startup` / `.axis-load` / `.axis-resp` / `.axis-cmplx` | small categorical chips |
| `.code` + `.cmt` / `.kw` / `.str` / `.dup` / `.new` | syntax-tinted code blocks; `.dup` highlights repetition, `.new` highlights an addition |
| `[data-slider]` + `.slider-*` | a draggable before/after **image** overlay (wire markup per `report.js`) |
| `.app-surface` | a dotted "canvas" backdrop for embedding a real-component demo |

Diagrams: prefer **inline SVG or styled HTML/CSS boxes** (arrows, lanes, flow steps) over images — they theme correctly in light/dark and stay crisp. For an interactive walkthrough (e.g. "trace a request through the layers"), add a small page-scoped `<script>` after the content; keep it dependency-free.

### Debug overlay — name every part (when the report plans a UI)

If a page embeds a UI demo or mockup, **tag its parts so the user can point at them precisely.** Iterating over chat is far faster when every region has a name both sides can say — "make the *depth-score* bigger", "move *rating* next to *flag*" beats "the thing under the title". `report.js` already provides the overlay; you just tag:

- Leaf, content-bearing parts → `data-dbg="<name>"` (e.g. `title`, `rating`, `flag`).
- Structural groups → `data-dbgc="<name>"` (e.g. `row`, `action-bar`, `tags`).
- Name parts **the way the user would say them**, not the CSS class. Don't tag meaningless wrappers — an empty labelled box just confuses; label the meaningful thing inside.

A floating **🔍 Debug overlay** toggle (bottom-right) appears automatically **only when a page has tagged parts**, and `?dbg` in the URL force-enables it (handy for screenshots). Labels render in a JS top layer, so they survive `overflow:hidden` clamps/rounded thumbnails that would clip a CSS pseudo-element. Screenshot the overlay-**on** version alongside the normal one — the labelled shot becomes the shared map for every later "change X" request.

**Make it interactive — the user strongly prefers live, playable demos over static figures.** A report site reads better when the reader can *do* something: trace a request by stepping through layers, toggle a before/after, fire a real component on the `.app-surface` canvas and watch it animate, drag a knob and see the result update. Wherever a concept has behavior (motion, state, a flow, a tradeoff with a dial), embed a **live playground** — real controls (buttons, toggles, segmented switches, sliders) wired with dependency-free page-scoped JS — instead of (or alongside) a static screenshot. "Self-contained / no build step" means *no framework or bundler*, **not** "no JavaScript"; lean into vanilla JS for interactivity. Keep static figures for at-a-glance scanning, but add the interactive layer by default whenever there's something to play with, and gate motion behind `prefers-reduced-motion`.

## Step 4 — build the CSS, then serve

The vendored `assets/tailwind.css` only contains the utility classes present when it was built. After you've written the real pages, **regenerate it so every class you used is included**:

```bash
cd "$DEST" && ./build-css.sh     # rescans all .html here, rewrites assets/tailwind.css (offline; uses a cached CLI in /tmp)
./serve.sh                       # then open the printed URL
```

If `build-css.sh` can't install the CLI (no network, nothing cached), the bundled base `tailwind.css` still covers the common utilities — the page won't be broken, just possibly missing a few exotic classes. Prefer running it when you can.

`./serve.sh` runs **this one site** in the foreground (fine for a quick local look). For the persistent link the user keeps open, **do not** start a per-site pm2 process — register the site with the project's single dashboard server instead:

```bash
"$PLANS_ROOT/_server/ensure.sh"   # start-or-reuse the project dashboard; prints its URL
```

`ensure.sh` is idempotent: it runs the dashboard under pm2 as `<project>-plans` on a deterministic per-project port, **reusing it if it's already up** (no duplicate processes, no port sprawl). Because discovery is dynamic, the new site shows up on the dashboard immediately — nothing to rebuild.

Then share the **hostname** URL it prints (`http://<host>:<port>/`), not `localhost` — the user is usually on a different machine. To point them straight at this site, deep-link with `http://<host>:<port>/<YYYY-MM-DD-slug>/`.

## Authoring principles

1. **One concern per page.** `index.html` carries the thesis/overview; each later page goes deep on one part. Don't sprawl one page across unrelated topics — that's what the multi-page sidebar is for.
2. **Lede first, then detail.** Every page opens with a kicker + headline + one-paragraph lede a busy reader can stop after. Put the "so what" up top, the mechanics below.
3. **Plain English, then precision.** Explain the idea the way you'd say it aloud, *then* pin it down with the exact mechanism, numbers, and `file:line` evidence. A reader should be able to follow the prose without the code and still trust it because it's cited.
4. **Pin claims to evidence.** Use `.evi` chips with real `file:line` references for factual/architectural claims and audit findings. An unsourced assertion in a report reads as a guess.
5. **Recommend, with a reason.** If the report proposes options or surfaces issues, take a position — rank, recommend, and justify in a line. Don't hand the reader an undifferentiated pile.
6. **Show, don't just tell.** Numbers go in `.mbar`/`.delta`/`.statnum`, flows go in SVG/CSS diagrams, severities in `.sev` dots. A report is a *visual* medium — that's the whole reason it isn't a markdown file.
7. **Honest, calibrated severity.** For audits: most real findings are low/cleanup, not high. Call a sound system sound. Include a short "what looks wrong but isn't" section to show the review was rigorous — it builds trust more than a long list of nits.
8. **Real content, never lorem ipsum.** Use the project's actual names, paths, numbers, and states.

## README — required

Fill in `README.md`: a one-paragraph "what this is", the `./serve.sh` view instructions, a page-by-page table, and a **provenance** section (where the content came from — measured evidence, prior plans/handoffs, consults). The README is how the next person (or you, later) knows what the site claims and how much to trust it.

## After building

- **Regenerate the CSS** (`./build-css.sh`) so the vendored stylesheet matches the pages.
- **Write `meta.json`** with the real title + one-line description so the dashboard card is accurate (not the `<…>` placeholder).
- **Register it with the project dashboard** (`plans/_server/ensure.sh`, see Step 4) — one server hosts all this project's plans and plan-sites — and give the user the **hostname** dashboard URL (or a deep-link to the new site), not `localhost`. Don't spin up a per-site pm2 process; the link keeps working after the session ends because the dashboard persists under pm2.
- If you can render headlessly, **screenshot a page or two** to verify layout (light + dark) before declaring it done. A report that looks broken undermines its content.
- This is a **read deliverable**: it changes no app code. Build it, surface it, and let the user read — don't act on its proposals until they say to.

## What to avoid

- **A CDN dependency.** The bundle must open and render offline. Tailwind is vendored; don't add `<script src="https://cdn…">`.
- **Forgetting `data-toc` (or `id`) on a section.** The sidebar's "on this page" list and scroll-spy are built from `[id][data-toc]` elements — a section without both simply won't appear in the nav.
- **Hand-typing page numbers or a sidebar list into the HTML.** That's the old bug. Pages, numbers and groups come from `site-nav.js` only; if a page is mis-numbered or missing, fix the manifest, not the markup.
- **A `site-nav.js` whose `file:` doesn't match the real filename.** The current-page highlight matches on filename — a typo there silently un-highlights the page.
- **A wall of prose with no visuals.** If a page has no diagram, bar, table, or chip, it should probably be a markdown `plan`, not a `plan-site`.
- **Unsourced claims.** Pin them with `.evi` or don't make them.
- **One giant page.** Split concerns across pages; the sidebar exists to carry them.
- **Editing the vendored `tailwind.css` by hand.** Change classes in the HTML and rerun `build-css.sh`.
- **A per-site pm2 server.** One dashboard per project hosts every site and plan — use `plans/_server/ensure.sh`, don't `pm2 start` each site on its own port.
- **A missing or placeholder `meta.json`.** Without it the dashboard falls back to scraping; with a stale `<Report Title>` placeholder the card is wrong. Fill it in.
